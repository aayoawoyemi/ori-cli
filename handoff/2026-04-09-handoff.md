## What This Is
Aries CLI (aka Ori CLI) — a memory-native coding agent running in the terminal. Built on top of Claude API with a custom harness that owns the cognition loop. The model is the muscle; the harness is the brain.


## Current Mode System

### Permission Modes (Alt+M to cycle in UI)
- `default` — asks permission for write ops
- `accept` — auto-accepts edits, still asks for Bash
- `plan` — agent can only write to the plan file
- `research` — research pipeline mode
- `yolo` — everything auto-approved

### Tool Phases (automatic, not user-facing)
- `lean` — starts here. REPL-enabled: `Repl, Edit, Write, Bash, VaultAdd, ProjectSave, WebFetch, WebSearch`
- `full` — widens automatically if model requests a tool not in lean set

**The problem:** There's no user-controllable *task mode* that constrains tool schemas per activity type. The phase tracker only widens (lean → full), never narrows. Permission modes control *approval*, not *visibility*.

## The Task Mode Feature (To Build)

Research-backed priority from SWE-agent ACI findings: schema filtering (removing tools the model can't see) beats verbal instruction every time. 10.7pp improvement from the same model just by constraining the action space.

### Proposed Task Modes (Alt+Z to cycle)

| Mode | Tools Visible | Use Case |
|------|--------------|----------|
| **Explore** | `Repl` only (codebase.*, vault.*) | Understanding code, reading files, searching memory |
| **Code** | `Repl, Edit, Write, Bash` (build/test only) | Making changes, running tests |
| **Research** | `Repl, WebFetch, WebSearch, VaultAdd` | External info gathering, paper reading |
| **Full** | Everything | No constraints, backwards-compatible |

### Key Design Decisions Needed
1. **Alt+Z keybinding** — separate axis from Alt+M permission modes
2. **Mode awareness in system prompt** — model should know what mode it's in and refuse out-of-mode requests gracefully ("Switch to Code mode for that")
3. **Status bar display** — show both permission mode AND task mode
4. **Soft vs hard** — should out-of-mode tool requests error, or should they trigger a mode-switch suggestion?

### Research Supporting This
- SWE-agent ACI: same model, better middleware = 10.7pp improvement
- Claude Code filters tool schemas per sub-agent
- OpenDev paper (arXiv 2603.05344): "LLM never sees tool definitions it cannot use"
- arXiv 2505.18135: Small description edits yield up to 10x change in tool usage frequency
- ToolMem (arXiv 2510.06664): Per-tool capability summaries across sessions = 14.8–28.7% better selection
- Token cost: failed tool calls are 3x cost (output + error + retry). 60-80% of agent tokens are waste (Elementor data)

## Soft Rewrite Layer (To Build)

Instead of blocking Bash and returning an error (current approach — maximum token waste), intercept and silently rewrite:

| Agent Requests | Harness Rewrites To | Notes |
|---------------|---------------------|-------|
| `bash: grep "pattern" file.ts` | `repl: codebase.search("pattern")` | Return structured results |
| `bash: cat src/loop.ts` | `repl: codebase.get_context("src/loop.ts")` | Full file content |
| `bash: find . -name "*.ts"` | `repl: codebase.list_files()` | Filtered listing |
| `bash: ls src/` | `repl: codebase.list_files()` | Directory listing |

Implementation point: `BashTool.execute()` or a PreToolUse hook. The model sees the result it wanted, learns the pattern. Zero wasted tokens.

## Config

Config file: `~/.aries/config.yaml` (or `~/.ori/config.yaml`)
Types: `src/config/types.ts`
Defaults: `src/config/defaults.ts`

Key settings:
- `repl.enabled: true` — turns on Python body, constrains Bash, enables codebase.* tools
- `signature.codebase.level` — controls ambient codebase signature density
- `preflight.enabled: 'auto'` — auto-disables when REPL is on (model pulls memory on-demand)
- `compact.threshold: 0.8` — auto-compaction at 80% context window

## Mini-SWE-agent Caveat

Research note: as frontier models improve, heavy tool interception has diminishing returns. Build the interception layer with a dial, not a wall. The `stripNavigationTools()` function already exists but is unused — it's the nuclear option. Task modes are the graduated approach.
